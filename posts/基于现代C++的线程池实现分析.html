基于现代C++的线程池实现分析

1. 概述

本文将从线程的基本概念出发，先说明一个普通的 Hello World 程序实际上运行在主线程中，再通过 std::async 让同样的输出逻辑异步执行。理解这一点之后，再进一步讨论为什么工程中不能总是为每个任务临时创建线程，以及线程池如何通过「固定工作线程 + 任务队列」来复用并发执行能力。

后文会基于一个开源的 C++ 线程池实现，依次分析现代 C++ 中与线程、可调用对象、类型擦除、future、packaged_task、可变参数模板和完美转发相关的内容。也就是说，线程池是本文的工程对象，而现代 C++ 的这些语言/库机制，是支撑这个对象成立的基础。

2. 初识线程池

2.1 线程

大多数初学者都会在一些最经典的计算机组成原理和操作系统课程中，看到下面这样类似的关于进程与线程的概念描述：

进程是系统分配资源的最小单位，线程是CPU调度的最小单位。

这句话给出了进程与线程的职责边界：进程更偏向资源的组织方式，线程更偏向代码的执行方式。多个线程可以存在于同一个进程中，共享同一份地址空间和进程资源，但它们各自拥有独立的执行栈、寄存器上下文和调度状态。

这些概念本身并不复杂，但对应到 C++ 代码时，初学者很容易被各种诘屈聱牙的接口绕进去。因此我们先不急着看线程池，而是从最熟悉的 Hello World 开始，观察一段最普通的 C++ 代码到底运行在哪个线程中。

​x
#include <iostream>
#include <thread>


int main()
{
    std::cout << "Hello World by " << std::this_thread::get_id() << std::endl;
    return 0;
}

初学 C++ 时，我们通常会把注意力放在 main 函数、std::cout 和程序输出上，但从操作系统的视角看，这段代码并不是脱离线程执行的。程序启动后，操作系统会先创建一个进程，为它分配地址空间、文件句柄等资源；随后在这个进程中启动一个初始执行流，也就是通常所说的主线程（Main Thread）。main 函数正是运行在这个主线程中的。

代码中的 std::this_thread::get_id() 用于获取当前代码所在的线程 ID。上面的程序可能输出如下结果：

xxxxxxxxxx
Hello World by 0x1f2e9de80

不同平台打印出的线程 ID 格式可能不同，但这里需要关注的是：普通的 Hello World 并不是一个“没有线程”的程序，它只是默认运行在主线程中。

2.2 可调用类型

上面的 Hello World 可以理解为：主线程执行了一次输出任务。接下来如果希望这次输出由异步任务执行，首先要把“输出”这件事从 main 函数中抽出来，变成一个可以被线程接口接收和调用的对象。

直观地说，如果存在一个用于启动任务的伪接口，我们希望调用形式类似这样：

xxxxxxxxxx
run_in_another_thread(say_hello);

这里的 say_hello 就需要是一个可调用对象（Callable）。通俗地说，如果一个对象 A 可以写成 A() 或 A(x) 这样的函数调用形式，那么它就可以被视为可调用对象。普通函数、lambda 表达式、重载了 operator() 的类对象，都属于这个范畴。

因此可以先把输出逻辑改写成一个普通函数：

xxxxxxxxxx
#include <iostream>
#include <thread>


void say_hello()
{
    std::cout << "Hello World by " << std::this_thread::get_id() << std::endl;
}


int main()
{
    say_hello();
    return 0;
}

此时 say_hello 就是一个普通函数，也可以作为可调用对象传给线程相关接口。更一般地，C++ 中常见的可调用对象包括：

这里先建立一个基本印象：线程接口并不是接收一段散落的语句，而是接收一个可以被调用的任务。后文分析任务队列时，std::function<void()> 正是用来统一保存这些不同形式的可调用对象。

2.3 第一个多线程程序

从 C++11 开始，标准库提供了 <thread>、<future> 等并发设施。最直接的线程对象是 std::thread，但为了先理解“提交一个任务并等待它完成”这个过程，可以从 std::async 入手：传入一个可调用对象，标准库负责启动一次异步任务，并返回一个 std::future 用于等待结果。

下面把前面的 Hello World 改写成异步执行版本。为了便于观察，我们先在主线程中调用一次 say_hello，再通过 std::async 启动一次异步任务：

xxxxxxxxxx
#include <future>
#include <iostream>
#include <thread>


void say_hello()
{
    std::cout << "Hello World by " << std::this_thread::get_id() << std::endl;
}


int main()
{
    say_hello();
    std::future<void> ret = std::async(std::launch::async, say_hello);
    ret.get();
    return 0;
}

上面这段程序可能输出如下结果，可以看到两次输出对应不同的线程 ID：

xxxxxxxxxx
Hello World by 0x1f2e9de80
Hello World by 0x16b373000

其中第一次 say_hello() 是普通函数调用，仍然运行在主线程中；第二次 say_hello 被传给 std::async，由标准库启动的异步任务执行。

这里显式写出 std::launch::async，是为了要求标准库立即异步执行这个任务。如果省略该策略，std::async 也可能选择延迟执行，即直到调用 get() 时才执行任务。

最后的 ret.get() 用于等待异步任务结束。对于有返回值的任务，get() 会取回任务的返回值；本例中的 say_hello 返回 void，所以 std::future<void> 只承担同步等待的作用。后文会继续展开 future、packaged_task 和线程池任务返回值之间的关系。

2.4 池化

std::async 很适合用来理解一次异步执行，但在工程场景中，任务通常不是只有一个，而是持续不断地产生。如果每来一个任务就创建一个线程、任务结束再销毁线程，那么线程的创建与销毁开销会非常可观，并且线程数量也可能随着任务数量失控增长，造成大量的上下文切换。线程池（Thread Pool） 正是为了解决这一问题：它在程序启动时预先创建固定数量的工作线程，并维护一个任务队列。外部代码只负责提交任务，工作线程不断从队列中取出任务执行，执行完毕后不销毁，而是继续等待下一个任务。

因此，在本质上我们可以把线程池看成是一个生产者-消费者模型：外部线程作为生产者向任务队列提交任务，池内的工作线程作为消费者从队列中取出任务执行。其整体的工作流程参考如下

本文分析的实现源自开源项目 progschj/ThreadPool，它用不到 90 行代码实现了一个支持获取返回值的通用线程池。代码示例如下

{
public:
    ThreadPool(size_t);

        -> std::future<typename std::result_of<F(Args...)>::type>;
    ~ThreadPool();
private:
    std::vector< std::thread > workers;        // 工作线程组
    std::queue< std::function<void()> > tasks; // 任务队列


    std::mutex queue_mutex;                     // 保护任务队列的互斥锁
    std::condition_variable condition;          // 任务到来/停止的通知
    bool stop;                                  // 停止标志
};

3. std::function 与类型擦除

本节对应源码中的任务队列 std::queue<std::function<void()>>。2.2 节已经介绍了 C++ 中常见的可调用类型，本节进一步说明三个问题：任务队列为什么选择 std::function、它背后的类型擦除是如何实现的、这一机制的优势与代价分别是什么。

std::bind 本身也是可调用对象相关的标准库工具，但它属于模板技巧的范畴，留到 5.4 节介绍，本节聚焦 std::function。

3.1 为什么任务队列选择 std::function

线程池的任务队列需要存放的，是一批「能被调用」的任务。问题在于它们的类型各不相同：可能是 lambda、函数指针，也可能是 std::bind 的结果，无法用某个具体类型的容器统一保存。因此需要一个通用的可调用对象包装器，这正是 std::function 的用途。

std::function<R(Args...)> 能持有任何「调用签名匹配」的可调用对象

xxxxxxxxxx
std::function<void()> f;
f = []{ /* lambda */ };          // lambda
f = some_functor;                // 仿函数
f = &some_free_function;         // 函数指针
f();                             // 统一用 () 调用

源码把任务队列定义成 std::queue<std::function<void()>>，就是把所有任务统一成「无参、返回 void」的 std::function<void()>，工作线程不必关心任务原本长什么样，只要 task() 调用即可。

为什么签名是 void()？因为工作线程只做「执行任务」这一件事：它既不向任务传参（参数在提交时就已经绑好，见 5.4 节），也不关心返回值（返回值通过 future 单独取回，见 4.3 节）。所以对工作线程而言，任务统一成 void() 最简洁。

不过这里存在一个实现上的问题：lambda 的类型是编译器生成的，写不出名字，而且每个 lambda 都是各自独立的类型。std::function 要把这些类型各异、甚至叫不出名字的对象统一装进同一个类型里，依赖的机制是类型擦除（Type Erasure）。

3.2 类型擦除的实现

类型擦除的核心思路是把「对象本身」和「能对它做的操作」分开：用一个抽象基类声明统一的调用接口，再用一个模板派生类把具体的可调用对象包进去，对外只持有指向基类的指针。下面是一个只保留核心逻辑的极简版本

xxxxxxxxxx
// 一个极简的 std::function<void()>，只演示类型擦除的核心
class Function
{
    // 抽象基类：抹掉具体类型，只暴露「可被调用」这一行为
    struct CallableBase
    {
        virtual void call() = 0;
        virtual ~CallableBase() = default;
    };


    // 模板派生类：把某个具体类型 T 的可调用对象包在里面
    template<class T>
    struct Callable : CallableBase
    {
        T func;
        Callable(T f) : func(std::move(f)) {}
        void call() override { func(); }   // 转发给真正的对象
    };


    CallableBase* ptr;   // 只持有基类指针，并不知道 T 到底是什么


public:
    template<class T>
    Function(T f) : ptr(new Callable<T>(std::move(f))) {}   // T 在这里被推导并固定下来
    ~Function() { delete ptr; }


    void operator()() { ptr->call(); }   // 通过虚调用分派到具体类型
};

关键在构造函数：它是一个模板，传入什么可调用对象，T 就被推导成对应的类型，并 new 出一个 Callable<T>。具体类型 T 被「关」在了派生类模板里，而成员 ptr 只是一个 CallableBase*，外部看不到 T 的任何信息。调用 operator() 时，通过基类指针发起一次虚调用，跳转到 Callable<T>::call，再由它转发给真正的对象。整个调用链如下

这样一来，无论底层 T 是 lambda、函数指针还是仿函数，对外都只通过「基类指针 + 虚函数 call」来调用，具体类型的差异在 ptr 这一层被抹平了。回到线程池，队列里每个任务都是 std::function<void()>，工作线程统一用 task() 触发这条分派链，不需要知道任务原本是什么类型。这就是「不同任务都能用同一个 task() 调用」的底层原理。

3.3 优势与代价

标准库的 std::function 在上述骨架之上还做了小对象优化（Small Object Optimization, SOO）：它内部带一小块缓冲区，足够小的可调用对象（比如只捕获一两个指针的 lambda）会直接存在缓冲区里，省掉一次堆分配，只有较大的对象才退回到堆上。

把类型擦除的得失归纳一下

优势

  • 统一异构类型：把类型各异的可调用对象收敛成同一个 std::function 类型，于是能放进同一个容器（任务队列）。这正是线程池能成立的前提。

  • 像普通对象一样使用：std::function 可拷贝、可赋值、可作为参数和返回值传递，调用方完全不必知道底层的具体类型。

代价

  • 间接跳转：每次调用都要经过一次虚调用（或函数指针跳转），编译器难以将其内联。

  • 可能的堆分配：当可调用对象超出 SOO 的小缓冲区时，会退回到堆上分配。

因此 std::function 并不是零开销的抽象。但对线程池来说，任务本身的执行通常远比一次间接跳转昂贵，这点开销可以忽略。

还有一个容易忽略、却对线程池很关键的约束：std::function 自身可拷贝，因此它要求所存的可调用对象也必须可拷贝构造。而 6.3 节会看到，std::packaged_task 恰恰只可移动、不可拷贝，这正是它必须先套一层可拷贝的 shared_ptr 才能进队列的根本原因。

4. 多线程接口

在 C++11 之前，标准库里并没有线程，写多线程程序只能依赖平台相关的接口（POSIX 的 pthread、Windows 的线程 API）或者像 Boost.Thread 这样的第三方库，可移植性和易用性都比较差。从 C++11 开始，标准库正式纳入了一整套并发设施：<thread>、<mutex>、<condition_variable>、<future>、<atomic>，并在之后的 C++14/17/20 中持续扩展（如 std::shared_mutex、std::scoped_lock、std::jthread、std::latch 等）。

这一节把线程池要用到的几样工具过一遍，它们分别对应源码里的三处：workers（线程）、queue_mutex 加 condition（同步），以及 enqueue 的返回值 future。这里只做简单介绍，更系统的讲解可以参考 Anthony Williams 的 C++ Concurrency in Action。

4.1 std::thread：线程的创建与管理

std::thread 代表一个执行线程，构造时传入一个可调用对象，线程随即启动开始执行

xxxxxxxxxx
void worker() { /* ...... */ }


std::thread t(worker);   // 构造即启动
t.join();                // 等待线程执行结束

关于 std::thread，有几点需要特别留意，它们直接决定了线程池里 workers 的写法

  • 所有权与移动语义：一个 std::thread 对象独占地持有一个底层线程，因此它只可移动、不可拷贝。把线程对象放进容器（比如线程池里的 std::vector<std::thread>）时，依赖的正是移动语义。

  • join 与 detach 二选一：线程对象在析构前必须明确它的归宿，要么 join（阻塞等待它结束），要么 detach（与它脱离关系，任其在后台运行）。如果两者都没做，std::thread 的析构函数会直接调用 std::terminate 终止程序。这也是为什么线程池的析构函数一定要 join 所有工作线程。

  • 传参方式：传给线程的参数会被拷贝或移动到线程自己的存储中，如果要传引用，需要用 std::ref 包装。

  • 线程数量：std::thread::hardware_concurrency() 会返回硬件支持的并发线程数（通常是 CPU 核心数），常用来决定线程池开多少个工作线程比较合适。

4.2 互斥量与条件变量：在线程间共享数据

多个线程访问同一份数据时，需要用互斥量来避免数据竞争。std::mutex 提供加锁/解锁，但实践中很少直接调用 lock()/unlock()，而是用 RAII 包装器来保证异常安全

  • std::lock_guard：最简单的 RAII 锁，构造加锁、析构解锁，中途不能解锁。

  • std::unique_lock：更灵活，支持中途解锁、延迟加锁、所有权转移，也是配合条件变量时必须使用的类型。

当一个线程需要等待「某个条件成立」时（比如工作线程等待队列里出现任务），不停地轮询检查会白白消耗 CPU。std::condition_variable 让线程在条件不满足时阻塞挂起、让出 CPU，等到别的线程修改了条件并调用 notify_one/notify_all 时再被唤醒

xxxxxxxxxx
std::unique_lock<std::mutex> lock(mtx);
cv.wait(lock, []{ return ready; });   // 条件不满足则阻塞，满足才往下走

wait 带谓词（Predicate）的这种写法，等价于下面这段循环

xxxxxxxxxx
while (!pred())
    wait(lock);

也就是说，只有当谓词返回 false 时线程才会真正阻塞并释放锁；一旦被唤醒，它会重新检查谓词，只有谓词为 true 时才会继续向下执行。这样写能正确处理虚假唤醒（Spurious Wakeup）：条件变量可能在没有任何 notify 的情况下被操作系统唤醒，带谓词的 wait 会在唤醒后重新检查条件，不满足就继续睡。线程池的工作线程空闲时就是靠它挂起、有任务时被唤醒的，这一点会在下文结合源码再展开。

4.3 future / promise / packaged_task：等待一次性事件

互斥量和条件变量解决的是「持续地保护共享数据」，但很多时候我们要的是另一种东西：启动一个任务，然后在将来某个时刻拿到它的结果。这种「一次性事件」在现代 C++ 里用 std::future 来表示，它也是线程池里 enqueue 返回给调用者的东西。

std::future<T> 代表一个「将来才会有的值」。它和结果的生产者之间通过一个共享状态（Shared State）连接：生产者把结果（或异常）写入共享状态，消费者通过 future 把它取出来。

调用 future.get() 会阻塞，直到结果就绪后返回；如果任务执行时抛出了异常，异常会被存进共享状态，并在 get() 时重新抛出，这让跨线程的错误处理变得和普通函数调用一样自然。

标准库提供了三种产生 future 的方式，分别对应不同的抽象层次

xxxxxxxxxx
// 1) std::async：最省心，把任务直接交给标准库去跑
std::future<int> f1 = std::async([]{ return compute(); });


// 2) std::packaged_task：把「可调用对象」和「future」绑在一起，
//    但何时、在哪个线程执行由你自己决定
std::packaged_task<int()> task([]{ return compute(); });
std::future<int> f2 = task.get_future();
std::thread(std::move(task)).detach();   // 比如丢到另一个线程去执行


// 3) std::promise：最底层，手动设置结果
std::promise<int> p;
std::future<int> f3 = p.get_future();
p.set_value(42);                         // 在某处手动写入结果


int r = f1.get();   // get() 阻塞直到结果就绪

5. 模板技巧

源码里 enqueue 的签名集中了好几个模板特性

xxxxxxxxxx
template<class F, class... Args>
auto enqueue(F&& f, Args&&... args) 
    -> std::future<typename std::result_of<F(Args...)>::type>
{
    // ......
    std::bind(std::forward<F>(f), std::forward<Args>(args)...)
    // ......
}

它要同时做到三件事：接受任意签名的任务、推导出任务的返回值类型、再把带参调用打包成 packaged_task 需要的无参形式。本节依次介绍其中用到的模板技巧，为第 6 节阅读 enqueue 做准备。

5.1 可变参数模板：接受任意签名的任务

为了支持任意的任务函数，enqueue 使用了可变参数模板（Variadic Templates）

xxxxxxxxxx
template<class F, class... Args>

其中 F 是可调用对象的类型，Args... 是一个参数包（Parameter Pack），可以匹配任意数量、任意类型的参数。形参 Args&&... args 则把这些参数一一接收下来，调用时再用 args... 把整个包展开。这样不管用户提交的任务是无参、还是带三五个参数，enqueue 都能用同一套模板接住。

5.2 转发引用与完美转发

注意 enqueue(F&& f, Args&&... args) 里的 &&，它在模板里不是右值引用，而是转发引用（Forwarding Reference，也叫万能引用）：当 F、Args 是待推导的模板参数时，F&& 既能绑定左值也能绑定右值，并把实参原本的左值/右值属性记录下来。

要把这个属性原样传递下去，需要配合 std::forward，这就是完美转发（Perfect Forwarding）

xxxxxxxxxx
std::forward<F>(f)              // 传入右值就以右值转发（触发移动）
std::forward<Args>(args)...     // 传入左值就以左值转发（触发拷贝）

它的意义是避免不必要的拷贝：如果调用者传进来的是个临时对象（右值），转发时就以右值身份继续传递，触发移动而不是拷贝；如果不用 forward，所有具名参数到了函数体里都会退化成左值，转发下去就只能拷贝，对于大对象（比如一个装满数据的 vector 任务参数）就是不必要的开销。5.4 节里 enqueue 把 f 和 args... 交给 std::bind 时，用的就是完美转发。

5.3 尾置返回类型与返回值推导

enqueue 的返回类型写在了参数列表后面，这叫尾置返回类型（Trailing Return Type）

xxxxxxxxxx
auto enqueue(...) -> std::future<typename std::result_of<F(Args...)>::type>;

之所以要写成尾置，是因为返回类型依赖模板参数 F 和 Args，必须等参数列表「就位」之后才能表达。其中 std::result_of<F(Args...)>::type 表示「用 Args... 调用 F 所得到的返回值类型」，于是整个函数返回的就是一个 std::future<任务返回值类型>。

std::result_of 在 C++17 中已被标记为弃用，并在 C++20 中被移除，官方推荐使用语义更清晰的 std::invoke_result 替代。如果在较新的标准下编译，可以写成 std::invoke_result_t<F, Args...>。

5.4 std::bind：打包成无参可调用对象

std::packaged_task 在线程池里被实例化成 packaged_task<return_type()>，注意括号里是空的，也就是它要求一个无参可调用对象。可用户提交的任务往往是带参数的，两者如何对接？这就是 std::bind 要解决的问题。

std::bind 的作用是把一个可调用对象和它的部分（或全部）参数预先绑定在一起，生成一个新的可调用对象

xxxxxxxxxx
auto bound = std::bind(f, arg1, arg2);   // 把 f 和它的参数绑在一起
bound();                                 // 调用时不用再传参，等价于 f(arg1, arg2)

绑定之后，原本 f(arg1, arg2) 这样的带参调用，就变成了 bound() 这样的无参调用，正好满足 packaged_task<return_type()> 的要求。源码里写的是 std::bind(std::forward<F>(f), std::forward<Args>(args)...) 也就是在绑定的同时，对 f 和每个参数都做了 5.2 节讲的完美转发。

至此，我们已经对源码用到的工具有了初步了解：可调用对象与类型擦除、多线程接口、模板技巧。下面回到源码本身来接着分析。

6.1 类结构设计

线程池的数据成员可以分为「资源」与「同步」两部分，资源部分负责存放工作线程与任务，同步部分负责协调多个线程对共享任务队列的访问。

公开接口则只有三个

6.2 构造函数：工作线程的启动

构造函数的职责是启动指定数量的工作线程，每个线程都执行同一段「取任务-执行任务」的循环逻辑。

xxxxxxxxxx
inline ThreadPool::ThreadPool(size_t threads)
    :   stop(false)
{
    for(size_t i = 0;i<threads;++i)
        workers.emplace_back(
            [this]
            {
                for(;;)
                {
                    std::function<void()> task;
                    {
                        std::unique_lock<std::mutex> lock(this->queue_mutex);
                        this->condition.wait(lock,
                            [this]{ return this->stop || !this->tasks.empty(); });
                        if(this->stop && this->tasks.empty())
                            return;
                        task = std::move(this->tasks.front());
                        this->tasks.pop();
                    }
                    task();
                }
            }
        );
}

6.2.1 线程的创建

构造函数通过一个循环调用 workers.emplace_back(...)，每次都向 vector 中原地构造一个 std::thread，并传入一个 lambda 作为线程的执行体。这里用 emplace_back 而非 push_back，正是因为 std::thread 只可移动、不可拷贝，原地构造可以避免临时对象的产生。lambda 以 [this] 捕获了线程池对象的指针，从而能在线程内部访问 queue_mutex、tasks、stop 等成员。

6.2.2 任务等待循环

每个工作线程的主体是一个 for(;;) 无限循环，循环体内的逻辑可以拆解为「加锁等待 → 判断退出 → 取出任务 → 释放锁 → 执行任务」，其状态流转如下

其中最关键的一步是 condition.wait 的调用

xxxxxxxxxx
this->condition.wait(lock,
    [this]{ return this->stop || !this->tasks.empty(); });

这正是前文所讲的「带谓词的 wait」。谓词是 stop || !tasks.empty()，意思是「线程池停止了，或者队列里有任务」。只要这两个条件都不满足，线程就阻塞并释放锁、让出 CPU；一旦被 notify 唤醒，它会重新检查谓词，从而既能正确处理虚假唤醒，又保证工作线程只在「该干活」或「该退出」时才被放行，空闲时不空转。

被唤醒并通过谓词检查后，线程还要再判断一次是否应当退出

xxxxxxxxxx
if(this->stop && this->tasks.empty())
    return;

注意退出条件是 stop && tasks.empty()，即只有当线程池已停止、并且队列中的任务也已全部处理完时，线程才会退出。这样即使在析构过程中，已经提交的任务也会先被执行完，而不会被直接丢弃。

6.2.3 临界区的范围控制

这里取任务和执行任务被特意拆成了两步

xxxxxxxxxx
{
    std::unique_lock<std::mutex> lock(this->queue_mutex);
    // ...... 取出任务
    task = std::move(this->tasks.front());
    this->tasks.pop();
}   // lock 在此处析构，自动释放锁


task();  // 在锁之外执行任务

std::unique_lock 采用 RAII 机制管理锁，构造时加锁、析构时自动解锁。这里用一对大括号把加锁、取任务的过程限制在一个独立的作用域内，使得锁在取出任务后立即被释放，而真正耗时的 task() 调用则放在锁之外执行。

原因是：互斥锁保护的只是「任务队列」这一共享资源，任务本身的执行并不需要持有锁。如果把 task() 也放在锁内，那么一个线程在执行长耗时任务期间会一直占用 queue_mutex，其他工作线程将完全无法取出任务，线程池就退化成了串行执行。因此尽可能缩小临界区范围是并发编程中的一个重要原则。

6.3 enqueue：任务提交

enqueue 是整份实现里逻辑最密的部分，我们将对他的执行步骤进行解析。

xxxxxxxxxx
template<class F, class... Args>
auto ThreadPool::enqueue(F&& f, Args&&... args) 
    -> std::future<typename std::result_of<F(Args...)>::type>
{
    using return_type = typename std::result_of<F(Args...)>::type;


    auto task = std::make_shared< std::packaged_task<return_type()> >(
            std::bind(std::forward<F>(f), std::forward<Args>(args)...)
        );


    std::future<return_type> res = task->get_future();
    {
        std::unique_lock<std::mutex> lock(queue_mutex);
        if(stop)
            throw std::runtime_error("enqueue on stopped ThreadPool");
        tasks.emplace([task](){ (*task)(); });
    }
    condition.notify_one();
    return res;
}

第一步，确定返回类型。 函数签名用了可变参数模板接收任意任务、尾置返回类型加 std::result_of 推导出任务的返回值类型，并在函数体里用 using return_type = ... 起了个别名，方便后面复用。

第二步，把任务打包成 packaged_task。

xxxxxxxxxx
auto task = std::make_shared< std::packaged_task<return_type()> >(
        std::bind(std::forward<F>(f), std::forward<Args>(args)...)
    );

由里向外看：先用 5.4 的 std::bind 配合 5.2 的完美转发，把带参的 f(args...) 绑成一个无参可调用对象；再用它构造一个 std::packaged_task<return_type()>（一如前文所讲，这是专为「定义任务、稍后再执行」准备的工具）。

至于外面为什么还要套一层 std::make_shared，是因为任务队列是 std::queue<std::function<void()>>，而 std::function 要求所存对象可拷贝，偏偏 std::packaged_task 只可移动、不可拷贝，没法直接进队列。于是用可拷贝的 shared_ptr 持有它，再让入队的 lambda 按值捕获这个 shared_ptr：shared_ptr 可拷贝，lambda 就满足了 std::function 的要求；引用计数也顺便保证了 packaged_task 在执行前不会被提前销毁。

第三步，取出 future。

xxxxxxxxxx
std::future<return_type> res = task->get_future();

通过 packaged_task::get_future() 拿到与之关联的 future（见 4.3 的共享状态模型），它最终会被返回给调用者。注意这一步在任务入队之前完成：一旦入队，任务随时可能被工作线程取走执行，所以必须提前把 future 拿到手。

第四步，加锁入队并唤醒。

xxxxxxxxxx
{
    std::unique_lock<std::mutex> lock(queue_mutex);
    if(stop)
        throw std::runtime_error("enqueue on stopped ThreadPool");
    tasks.emplace([task](){ (*task)(); });
}
condition.notify_one();

入队同样要加锁保护队列。入队前先检查 stop：如果线程池已经停止，再提交任务没有意义（永远不会被执行），于是抛异常拒绝。真正入队的是那个按值捕获 shared_ptr 的 lambda，它被 std::function<void()> 借助 3.2 的类型擦除装了起来；工作线程将来调用 task() 时，就会沿着类型擦除的分派链走到 (*task)()，触发 packaged_task 运行、把结果写进共享状态，于是这里返回的 res 就能 get() 到值了。

最后在锁外调用 condition.notify_one()，唤醒一个正在等待的工作线程。只唤醒一个就够了，因为一次 enqueue 只新增一个任务；放在锁外则是个常见优化，避免被唤醒的线程立刻又因为锁没释放而再次阻塞。

6.4 析构函数：工作线程的停止与回收

析构函数负责通知所有工作线程停止，并等待它们全部结束。

xxxxxxxxxx
inline ThreadPool::~ThreadPool()
{
    {
        std::unique_lock<std::mutex> lock(queue_mutex);
        stop = true;
    }
    condition.notify_all();
    for(std::thread &worker: workers)
        worker.join();
}

它的关闭流程分为三步：加锁设 stop → notify_all 唤醒所有线程 → 逐个 join。

加锁设置停止标志。 设置 stop 时同样加锁，因为它是被多个线程读写的共享变量。加锁不仅保证写入的原子性，更重要的是和工作线程里 condition.wait 的谓词检查形成正确的同步关系，确保 stop = true 这一修改对被唤醒的线程可见。

唤醒所有线程。 这里用 notify_all 而非 notify_one，因为关闭时要让所有正在阻塞的工作线程都醒来。它们醒来后重新检查谓词 stop || !tasks.empty()，此时 stop 已为真，于是全部通过；接着各自判断 stop && tasks.empty()，把队列里剩余的任务逐一取出执行完，待队列清空后才退出循环。

逐个 join 等待。 最后遍历所有工作线程并逐个 join。这一步是必须的，一个仍然 joinable 的 std::thread 在析构时会触发 std::terminate，所以必须等所有线程结束，才能安全地销毁线程池。

综合 6.2.2 的退出条件与本节的关闭流程可以看到，这个线程池在析构时会把已经提交到队列里的任务全部执行完再退出。如果想要「立即停止、丢弃未执行任务」的行为，需要在工作线程的退出判断里去掉 tasks.empty() 这个条件。

6.5 使用示例

最后看一个典型的使用场景：创建一个包含若干工作线程的线程池，向其提交一批带返回值的任务，再通过 future 依次取回结果。

xxxxxxxxxx
#include <iostream>
#include <vector>
#include <chrono>
#include "ThreadPool.h"   // 上文分析的线程池实现


int main()
{
    // 创建一个包含 4 个工作线程的线程池
    ThreadPool pool(4);
    std::vector<std::future<int>> results;


    // 提交 8 个任务，每个任务返回它计算的平方值
    for(int i = 0; i < 8; ++i)
    {
        results.emplace_back(
            pool.enqueue([i] {
                std::this_thread::sleep_for(std::chrono::milliseconds(100));
                return i * i;
            })
        );
    }


    // 依次取回每个任务的结果，future.get() 会阻塞直到对应任务执行完毕
    for(auto && result : results)
        std::cout << result.get() << ' ';
    std::cout << std::endl;


    return 0;
}   // pool 在此处析构，自动等待所有任务完成并 join 所有线程

程序输出为每个任务计算的平方值

xxxxxxxxxx
0 1 4 9 16 25 36 49

可以看到，调用方不需要关心线程的创建、调度与销毁，只要把任务交给 enqueue，再在需要结果时调用 future.get() 就可以了。线程的管理细节都被挡在了线程池内部，上层代码只关心任务本身，而这些内部细节，正是前面 3、4、5 节逐一铺垫过的那些工具。
