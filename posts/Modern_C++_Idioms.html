现代 C++ Idioms

1. 概述

在 C++ 中，很多问题并不是单纯依靠某一个语法点解决的，而是由语言特性、标准库设施、对象生命周期和编译期约束共同组成一种稳定写法。这类长期沉淀下来的写法通常称为 Idiom，也就是惯用法。

Idiom 的价值在于把经验性的工程约束固化到代码结构中。例如资源释放可以交给对象析构，接口约束可以交给编译期重载决议，运行时多态的一部分场景也可以被模板和静态分派替代。也就是说，idiom 并不是为了让代码看起来更“高级”，而是为了让资源语义、对象语义和接口边界更加稳定。

下面进入具体 idiom。

2. Idioms

RAII：用对象生命周期管理资源

基本思想

RAII（Resource Acquisition Is Initialization） 是 C++ 中最核心的惯用法之一。它的思想是：资源的获取与对象构造绑定，资源的释放与对象析构绑定。这样一来，只要对象生命周期正确，资源释放就会自动发生。

下面用 std::unique_ptr 和自定义 deleter 管理一个动态资源：

xxxxxxxxxx
struct Resource {
    int data;
};


struct ResourceDeleter {
    void operator()(Resource* resource) const
    {
        std::cout << "releasing resource\n";
        delete resource;
    }
};


using ResourceHandle = std::unique_ptr<Resource, ResourceDeleter>;

使用时只需要持有 ResourceHandle：

xxxxxxxxxx
int main()
{
    ResourceHandle resource{new Resource{42}};
    std::cout << "resource data: " << resource->data << '\n';
}

当 resource 离开作用域时，std::unique_ptr 的析构函数会自动调用 ResourceDeleter，最终释放资源。

工程意义

RAII 的关键价值在于异常安全。如果中间代码抛出异常，普通的 delete 很容易被跳过，而 RAII 对象会在栈展开过程中自动析构。

在工程实践中，文件句柄、锁、网络连接、GPU 资源、临时状态恢复等都适合用 RAII 表达。C++ 标准库中的 std::unique_ptr、std::shared_ptr、std::lock_guard、std::scoped_lock 都是 RAII 思想的体现。

Scope Guard：作用域退出时执行清理

基本思想

RAII 不一定只能管理“资源对象”，也可以管理一段需要在作用域结束时执行的清理逻辑。Scope Guard 就是这种写法：构造时接收一个 cleanup 函数对象，析构时自动执行它。

代码参考如下：

xxxxxxxxxx
template <class F>
class ScopeGuard {
public:
    explicit ScopeGuard(F cleanup)
        : cleanup_(std::move(cleanup))
    {
    }


    ScopeGuard(ScopeGuard&& other) noexcept
        : cleanup_(std::move(other.cleanup_))
        , active_(std::exchange(other.active_, false))
    {
    }


    ScopeGuard(const ScopeGuard&) = delete;
    ScopeGuard& operator=(const ScopeGuard&) = delete;
    ScopeGuard& operator=(ScopeGuard&&) = delete;


    ~ScopeGuard()
    {
        if (active_) {
            cleanup_();
        }
    }


    void dismiss()
    {
        active_ = false;
    }


private:
    F cleanup_;
    bool active_{true};
};


template <class F>
ScopeGuard(F) -> ScopeGuard<F>;

这里使用类模板参数推导，因此可以直接写：

xxxxxxxxxx
auto unlock_on_exit = ScopeGuard{[&locked] {
    locked = false;
    std::cout << "scope exit unlocks resource\n";
}};

当 unlock_on_exit 离开作用域时，析构函数会检查 active_，如果没有被 dismiss() 取消，就执行传入的 lambda。

移动语义与取消机制

Scope Guard 有两个容易忽略的点。

首先，它通常应该禁止拷贝。因为拷贝会让同一段 cleanup 逻辑被多个对象持有，最终可能执行多次。示例中通过 ScopeGuard(const ScopeGuard&) = delete 和 operator=(const ScopeGuard&) = delete 禁止了拷贝。

其次，它可以允许移动，但移动后原对象需要失效。示例中的移动构造函数使用 std::exchange(other.active_, false)，表示新对象接管清理责任，旧对象不再执行 cleanup。

在工程实践中，Scope Guard 适合表达“无论中间如何退出，都要恢复现场”的逻辑，例如临时修改全局状态后恢复、函数返回前解锁、失败时回滚局部操作等。Library Fundamentals TS 中的 std::experimental::scope_exit 以及不少项目里的 scope_exit 工具，本质上都是这一类惯用法的封装。

Copy-and-swap：统一拷贝赋值与异常安全

问题描述

当一个类拥有多个成员，或者持有动态资源时，拷贝赋值运算符需要处理自赋值、旧资源释放、新资源构造失败等问题。直接在 operator= 中逐个成员赋值，很容易留下中间状态。

Copy-and-swap 的做法是：先通过值传参构造出一个临时副本，然后与当前对象交换内容。临时对象析构时会带走旧资源。

代码参考如下：

xxxxxxxxxx
class Product {
public:
    Product& operator=(Product other)
    {
        swap(*this, other);
        return *this;
    }


    friend void swap(Product& lhs, Product& rhs) noexcept
    {
        using std::swap;
        swap(lhs.name_, rhs.name_);
        swap(lhs.serial_, rhs.serial_);
    }


private:
    std::string name_;
    int serial_;
};

这里 operator= 的参数是按值传入的 Product other。如果右侧是左值，会触发拷贝构造；如果右侧是右值，则可以触发移动构造。构造成功后再执行 swap，当前对象状态一次性完成替换。

异常安全

Copy-and-swap 的异常安全来自两个阶段：

  1. 构造 other。如果构造失败，当前对象没有被修改。

  2. 执行 swap。如果 swap 是 noexcept，则状态交换不会失败。

也就是说，它把赋值操作转化为“先构造新状态，再原子式替换旧状态”。

这种 idiom 适合需要强异常安全保证的值类型。但如果类的成员很多，或者 swap 成本不低，也需要评估它是否会引入额外开销。

PImpl：隐藏实现细节

基本思想

PImpl（Pointer to Implementation） 的目标是把类的接口和实现分离。头文件中只暴露一个前向声明的 Impl 类型，以及一个指向它的指针；具体成员、依赖和实现细节放到 .cpp 中。

PImpl 的基本结构如下：

xxxxxxxxxx
class Widget {
public:
    explicit Widget(std::string name);
    ~Widget();


    Widget(Widget&&) noexcept;
    Widget& operator=(Widget&&) noexcept;


    Widget(const Widget&) = delete;
    Widget& operator=(const Widget&) = delete;


    void draw() const;


private:
    class Impl;
    std::unique_ptr<Impl> impl_;
};

真正的数据成员在 Widget::Impl 中：

xxxxxxxxxx
class Widget::Impl {
public:
    explicit Impl(std::string name)
        : name_(std::move(name))
    {
    }


    void draw() const
    {
        std::cout << "drawing widget: " << name_ << '\n';
    }


private:
    std::string name_;
};

工程意义

PImpl 的主要价值有三点：

  • 降低编译依赖：实现细节改变时，不必让所有包含头文件的用户重新编译。

  • 隐藏 ABI 和私有成员：二进制接口更加稳定。

  • 减少头文件暴露内容：复杂依赖可以留在 .cpp 中。

它的代价也需要明确：多了一次间接访问，通常还会引入一次堆分配。因此 PImpl 更适合库接口、跨模块边界、编译依赖很重的类，而不是所有小类都需要使用。

CRTP：静态多态

基本形式

CRTP（Curiously Recurring Template Pattern） 的形式是让派生类把自身类型作为模板参数传给基类：

xxxxxxxxxx
template <class Derived>
class Animal {
public:
    void greet()
    {
        static_cast<Derived*>(this)->call();
        Derived::static_call();
    }
};


class Dog : public Animal<Dog> {
public:
    void call();
    static void static_call();
};

也就是说，基类 Animal<Derived> 在编译期知道最终的派生类类型，因此可以通过 static_cast<Derived*> 调用派生类接口。

与虚函数的区别

CRTP 常用于实现静态多态。与虚函数相比，它不需要 vtable，也没有运行时动态分派开销。调用关系在编译期确定，编译器也更容易内联。

但 CRTP 的缺点是类型必须在编译期确定，不适合需要运行时异构容器的场景。例如 std::vector<std::unique_ptr<Base>> 这种运行时多态，仍然更适合用虚函数。

Mixin：复用行为组件

基本思想

Mixin 是一种通过继承组合行为的写法。它通常不是为了表达“is-a”关系，而是为了给目标类型附加一组可复用能力。

首先是不可拷贝组件：

xxxxxxxxxx
class NonCopyable {
protected:
    NonCopyable() = default;
    ~NonCopyable() = default;


    NonCopyable(NonCopyable&&) noexcept = default;
    NonCopyable& operator=(NonCopyable&&) noexcept = default;


private:
    NonCopyable(const NonCopyable&) = delete;
    NonCopyable& operator=(const NonCopyable&) = delete;
};

继承它之后，目标类自然获得“不可拷贝、可移动”的语义：

xxxxxxxxxx
class Socket : private NonCopyable, public Printable<Socket> {
public:
    explicit Socket(std::string name);
    Socket(Socket&&) noexcept = default;
    Socket& operator=(Socket&&) noexcept = default;
};

其次是基于 CRTP 的 Printable：

xxxxxxxxxx
template <class Derived>
class Printable {
public:
    void print() const
    {
        std::cout << static_cast<const Derived&>(*this).to_string() << '\n';
    }
};

只要派生类提供 to_string()，就能复用 print()。

适用边界

Mixin 的优势是把横向能力拆成小组件，例如“可打印”“可比较”“不可拷贝”“引用计数”等。它比复制粘贴成员函数更容易维护。

但 Mixin 也容易过度使用。如果一个类同时继承很多行为组件，读者需要在多个基类之间跳转，理解成本会上升。因此在工程中使用 Mixin 时，应该保证每个组件的语义足够单一，并且接口契约清晰。

Copy-on-write：写时复制

基本思想

Copy-on-write（COW） 的核心思想是：拷贝时先共享底层数据，直到某一方需要写入时，才真正复制一份独立数据。

下面使用 std::shared_ptr<T> 保存底层对象：

xxxxxxxxxx
template <class T>
class CowPtr {
public:
    explicit CowPtr(T value)
        : value_(std::make_shared<T>(std::move(value)))
    {
    }


    const T& read() const
    {
        return *value_;
    }


    T& write()
    {
        detach();
        return *value_;
    }


private:
    void detach()
    {
        if (value_.use_count() != 1) {
            value_ = std::make_shared<T>(*value_);
        }
    }


    std::shared_ptr<T> value_;
};

读操作直接返回共享数据；写操作先调用 detach()。如果当前数据被多个对象共享，就复制一份新的数据，保证写入不会影响其他对象。

工程注意事项

COW 在读多写少、数据拷贝很重的场景中可能有价值。但它也有不少隐含成本：

  • shared_ptr 引用计数有原子操作成本；

  • 多线程写入需要额外同步；

  • 隐式共享可能让性能行为不直观；

  • 标准库字符串曾经有过 COW 实现，但现代 C++ 标准对迭代器、并发和修改语义的要求使它不再是通用选择。

因此 COW 更适合特定数据结构或业务对象，而不是默认应用到所有值类型。

Erase-remove：从顺序容器中删除元素

问题描述

在 std::vector、std::string 这类顺序容器中，删除满足条件的元素时，很多初学者会直接在循环中调用 erase。这种写法不仅容易写错迭代器更新逻辑，也可能造成多次移动元素，复杂度和可读性都不理想。

C++ 中常见的做法是 erase-remove idiom：先用 std::remove 或 std::remove_if 把需要保留的元素移动到前面，再调用容器的 erase 删除尾部无效区间。

代码参考如下：

xxxxxxxxxx
std::vector<int> numbers{1, 0, 2, 0, 3, 0, 4};


numbers.erase(
    std::remove(numbers.begin(), numbers.end(), 0),
    numbers.end());

对于条件删除，可以使用 std::remove_if：

xxxxxxxxxx
std::vector<std::string> names{"alice", "", "bob", "", "carol"};


names.erase(
    std::remove_if(names.begin(), names.end(), [](const std::string& name) {
        return name.empty();
    }),
    names.end());

为什么需要两步

std::remove 并不会真正改变容器大小。它只是在迭代器范围内重排元素，把不需要删除的元素移动到前面，并返回新的逻辑结尾：

xxxxxxxxxx
before: 1 0 2 0 3 0 4
remove 0
after:  1 2 3 4 ? ? ?
              ^
              new_end

也就是说，std::remove 处理的是“算法范围”，并不知道底层容器是否可以缩短。真正改变容器大小的是成员函数 erase。

因此 erase-remove idiom 的完整形式总是：

xxxxxxxxxx
container.erase(
    std::remove(container.begin(), container.end(), value),
    container.end());

在 C++20 之后，标准库提供了 std::erase 和 std::erase_if，可以把这个惯用法进一步封装起来。但理解 erase-remove 仍然很有价值，因为它揭示了 STL 中“算法操作迭代器范围，容器成员函数负责结构变化”这一设计边界。

std::addressof：获取真实地址

问题描述

C++ 允许类重载 operator&。一旦某个类型重载了取地址运算符，表达式 &obj 就不一定返回对象的真实内存地址，甚至可能无法访问。

下面构造一个 operator& 为 private 的类型：

xxxxxxxxxx
class NonAddressable {
public:
    using UselessType = void;


private:
    UselessType operator&() const {}
};

此时直接取地址会失败：

xxxxxxxxxx
// auto* invalid = &value; // error: operator& is private

如果需要绕过重载的 operator&，应该使用 std::addressof：

xxxxxxxxxx
NonAddressable value;
auto* ptr = std::addressof(value);

适用场景

std::addressof 在普通业务代码里不常见，但在泛型库代码中很重要。模板代码面对未知类型时，不能假设 &obj 一定具有内建语义，因此标准库容器、分配器、智能指针等底层设施更倾向使用 std::addressof 获取真实地址。

这个 idiom 的意义在于提醒我们：C++ 中看似基础的运算符，也可能被用户类型重载。泛型代码要尽量避免对类型行为做过强假设。

Deleted overload：限制接口参数

基本思想

有时一个函数只希望接受非常明确的类型，而不是依赖隐式转换。例如只接受 int，拒绝 float、char、double 等类型。

可以使用 deleted overload 实现这一约束：

xxxxxxxxxx
void accept_int(int value)
{
    std::cout << "accepted int: " << value << '\n';
}


template <class T>
void accept_int(T) = delete;

当调用 accept_int(1) 时，会匹配非模板版本；当调用 accept_int(1.0f) 或 accept_int('1') 时，模板重载更匹配，但该重载被标记为 = delete，于是编译期报错。

xxxxxxxxxx
accept_int(1);


// accept_int(1.0f); // error
// accept_int('1');  // error

工程意义

这种写法的价值在于把错误提前到编译期，并且错误位置非常明确。它适合用来禁止危险的隐式转换，例如：

  • 禁止整数和布尔值混用；

  • 禁止窄化转换；

  • 禁止某些临时对象绑定；

  • 限制 API 只接受指定类型。

在 C++20 之后，很多约束也可以用 concepts 表达。Deleted overload 更轻量，适合小范围的接口限制；concepts 更适合成体系的模板约束。

User-defined literal：自定义字面量后缀

基本思想

C++ 允许为字面量定义后缀，使某些值可以用更接近领域语言的方式表达。下面定义一个 _date 后缀：

xxxxxxxxxx
std::string operator""_date(const char* text, std::size_t length)
{
    return std::string{text, length};
}

使用时可以写成：

xxxxxxxxxx
auto date = "2021-11-11"_date;

虽然这个例子只是返回 std::string，但它展示了 user-defined literal 的基本形态。实际工程中可以进一步把字符串解析成日期类型、长度单位、角度单位、时间单位等。

适用边界

自定义字面量适合表达强领域语义，例如：

xxxxxxxxxx
auto timeout = 500_ms;
auto angle = 90_deg;
auto size = 128_KiB;

它的优点是让单位和类型绑定到字面量上，减少裸数字带来的歧义。但也要避免为了语法好看而滥用后缀。一个好的 literal suffix 应该让语义更明确，而不是制造新的记忆负担。
