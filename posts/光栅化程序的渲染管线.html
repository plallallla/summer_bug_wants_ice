光栅化程序的渲染管线

本文对光栅化程序的渲染管线进行介绍介绍，对一些固定的管线流程相对详细一点，因为计划后续补充可编程管线的详细内容。

1. 光栅化渲染概述

一个完整的光栅化程序是能够接受绘制数据输入，在屏幕上显示画面的程序。因此从输出结果来看，渲染的终极目标是一个位图（Bitmap），我们最终得到的是一个显示在屏幕上的图像画面，而屏幕上的画面由若干个像素组成，每个像素都有两个维度的性质

  • 位置

  • 颜色

也就是说，我们需要决定每个像素在哪里以及每个像素长什么样。这两个问题对应决定了渲染管线的两个重要阶段

  • 几何阶段（像素位置）

  • 着色阶段（像素颜色）

从输入上看，我们需要准备的输入数据也是需要为解决这两个问题服务，内容应该包括不限于以下内容

  • 绘制的目标在什么位置（几何）

  • 绘制的目标是什么形状（几何）

  • 绘制的目标有什么颜色（着色）

2. 应用程序阶段

应用程序阶段主要由CPU完成，他负责为GPU提供渲染所需的全部数据和指令。CPU在准备阶段通常需要处理的主要工作有以下两个部分

  • 数据组织并传输（Data）

  • 发出绘制调用（Draw Call）

2.1 数据组织并传输

CPU需要为GPU准备好绘制的数据，这里主要是针对当前搭建的场景，依据用户输入与场景特性进行调整，比如用户输入响应、资源加载、可见性剔除（Culling）、渲染排序（Render Sorting）等等，经过一系列处理与操作后，最终形成GPU所需要的数据内容，并申请GPU的显存空间，将数据通过GPU的PCIe总线传输到显存上。

2.2 发出绘制调用

绘制调用其实是CPU与GPU之间命令的一系列统称，也包括了非绘制命令的同步调用。对于同步调用来说，他的功能是告诉GPU顶点数据、着色器程序等内容的绑定情况，变量、纹理的设置情况等等。对于绘制调用来说，他主要通知GPU需要进行绘制，并且绘制的方式什么。因此在现代图像API中，同步命令更多的基于命令缓冲区（Command Buffer）来实现以减少 CPU-GPU 同步开销。同时，通过合理的组织与管理，减少Draw Call的调用频率也是性能优化中非常重要的课题。

2.3 性能优化

图形程序的性能往往受限CPU与GPU的PCIe总线通信的带宽与速度、GPU显存等因素，因此在应用程序阶段如何尽可能的减少GPU工作所需的数据与调用来达成性能优化也是至关重要，下面对两个应用程序阶段的两个重要的性能优化工作进行简单介绍。

2.3.1 可见性剔除（Culling）

可见性剔除是在CPU场景管理时，依据一些原则和计算直接放弃一些场景的绘制，从而可以减少不必要的Draw Call，降低GPU负载。在工程实践中，主要可以实现以下几种功能

2.3.2 渲染排序（Render Sorting）

渲染排序是指将场景中的多个元素通过不同的组合方式来组织数据，因此通过渲染排序，我们可以最小化 GPU 状态切换开销，提升吞吐量，他主要可以通过以下几种方式来实现

  • 按材质/纹理/着色器分组（减少状态切换）

  • 透明物体按深度从后往前排序（保证正确混合）

  • 不透明物体按深度从前向后排序（利于 Early-Z 优化）

  • 实例化（Instancing）批处理，合并相同Mesh的多个实例

3. 几何阶段

几何阶段主要是将三维模型的顶点从“模型空间”逐步变换到“屏幕空间”，并处理图元的可见性（裁剪），为后续的光栅化做准备。这一阶段包括了顶点着色器（Vertex Shader）和几何着色器（Geometry Shader）两种可编程着色着色器，另外还包括了可配置的裁剪阶段（Clipping）与固定的屏幕映射阶段（Screen Mapping）。

3.1 顶点着色器（Vertex Shader）

顶点着色器读取显存里的场景顶点数据，依据场景需求与实际情况对顶点进行转换。正常来说读取到的显存数据是模型空间中的数据，最终经过一系列转换后输出成屏幕裁剪时所需的齐次空间坐标数据。一般来说，转换与修改的流程参考如下

gl_Position = projection * view * model * vec4(aPos, 1.0);

这里最终输出的顶点 gl_Position（ DX 中的 SV_POSITION ）坐标是基于齐次空间的值，他是一个四维的坐标 (x,y,z,w) 其中的 w 坐标值在标准的透视矩阵中是视觉空间的深度值的负恒等映射 w=−zeye，这个 w 坐标值将用于后续的透视除法与线性差值。

除了顶点坐标以外，顶点着色器可以实现顶点属性（Attribute）数据的多种操作，如创建 / 修改 / 忽视。这些属性通常包括单不限于以下内容：

  • 位置

  • 法线、切线、副切线

  • 纹理坐标

  • 实例ID

  • 骨骼

几何着色器可以理解为图元的加工器，他接受的输入是一个图元，输出可以是任意个图元（类型可与输入不同）。他不是一个必须的着色阶段，而是可选的。就图元的加工来说，他可以实现以下功能

在几何着色器之前，可以启用曲面细分阶段（包括 Hull Shader、固定功能 Tessellator 和 Domain Shader），将输入的补丁（Patch）动态细分为更密集的三角形网格，从而在不增加原始模型顶点数的前提下丰富几何细节。

现代图像 API 基本都支持几何阶段对顶点的细分处理与几何处理，但是这样的处理也带来一些问题，比如性能不可预测，带宽压力大等等，而作为一个可选的编程阶段，在大多数的开发需求下，几何着色器都没有对应的使用场景。因此在 Apple 的 Metal 中，苹果明确移除了这个阶段，但同样可以用 Compute Shader + Indirect Drawing 的方案来实现同样的效果，这也逐渐成为现在的一种流行趋势。

3.3 裁剪（Clipping）

裁剪阶段是可配置的，他根据输出图元的齐次空间坐标来进行判断，如果图元完全在视锥体内则进入下一阶段，如果完全不在则剔除出去，如果一部分在则进行裁剪以保留视锥内的部分，并且计算出交点的齐次空间坐标。

顶点属性值的计算也需要考虑齐次坐标的 w 分量来实现，例如计算与 near 平面 z=−w(t) 的交点，解得 t ，那么新顶点应为 vnew=v0+t(v0−v1) ，这里的新顶点的计算值将包括 (x,y,z,w) 与所有其他属性。

裁剪在实现上由硬件直接进行，其标准的裁剪多边形的算法是 Sutherland-Hodgman 算法，该算法通过分割处理，逐边裁剪的方法来将多边形裁剪分解为一系列简单问题，从而将输入的多边形裁剪出一个平面。尽管他不可编程，但在一些API中还是定义了配置接口，留给用户自行定义裁剪平面（User Clip Planes）。

3.4 屏幕映射（Screen Mapping）

3.4.1 透视除法

裁剪后的数据坐标是依然是齐次空间坐标，要进行下一步的屏幕映射操作首先需要将他转化为归一化设备坐标（NDC），这里将进行如下的计算

(2)xndc=xw,yndc=yw,zndc=zw

如果顶点着色器中的投影操作是非线性的透视投影，那么从这里的除法开始，屏幕空间的坐标讲也都成为非线性的。因此还需要保留 1w 的值用于后续的透视校正插值计算。

归一化除法导致非线性的原因：在顶点着色器（3.1）中提到过 w 坐标值在标准的透视矩阵中是视觉空间的深度值的负恒等映射 w=−zeye，因此这里的NDC坐标经过除法后，如果对 x 进行求导可得器变化率∂f∂x=1w=−1zeye，y 坐标同理。这意味着 NDC 空间下坐标的横纵坐标变化快慢与物体的深度有关，包括后续的屏幕空间的坐标也是非线性的，也就是近处物体的移动在屏幕上更明显，而远处物体的移动则相对变化不大。这一行为符合透视原理，也符合人眼的直觉。

3.4.2 屏幕空间映射

接下来对于NDC坐标的 xndc 与 yndc ，需要将这里的对应的值转为屏幕空间的坐标值，转换过程如下（NDC坐标以 [−1,+1] 为例）计算参考如下

(3)xscreen=(xndc+1)×width2+xoffsetyscreen=(yndc+1)×height2+yoffset

屏幕映射的执行代码参考如下

xxxxxxxxxx
// 视窗矩阵

{
    auto x = width / 16.;
    auto y = height / 16.;
    auto w = width * 7. / 8;
    auto h = height * 7. / 8;
    return

        w/2.,0.,0.,x+w/2.,
        0.,h/2.,0.,y+h/2.,
        0.,0.,1.,0.,

3.4.3 深度值计算

因为顶点着色器中透视投影矩阵的深度变换不是线性的，计算参考如下

(4)zclip=2nff−n×1zeye−f+nf−n

这里可以看到 zndc∝1zeye ，这将导致深度精度在空间上是不均匀的，也计算近处深度非常高而远处的精度却非常低，此时远处的物体容易发生远处物体容易发生 Z-Fighting。因此也存在一些需求使用均匀深度，此时的投影矩阵为正交投影，可以得到线性的深度值映射。但无论是不是线性的变换，屏幕映射对深度的操作都是一致的，此处只执行线性缩放和平移，参考如下

(5)zscreen=zndc×f−n2+f+n2

屏幕空间的深度值可以在片元着色器中通过 gl_FragCoord.z 获取，顶点着色器中投影变换线性与否决定了这里的深度值线性与否

3.4.4 顶点属性值

屏幕映射阶段不对任何顶点属性做任何处理，裁剪阶段的顶点属性值直接输出到下一阶段。

3.4.5 屏幕空间裁剪

坐标转换到屏幕空间后，我们还需要再对三角形进行一次判断，根据不同需求，有一些三角形可能不需要绘制。因此我们还需要做一次机遇屏幕空间坐标的裁剪操作来剔除那些不需要绘制的三角形。一般使用有向面积法来实现，假设已知三角形的三个顶点 A(x1,y1),B(x2,y2),C(x3,y3)，那么可以计算到他的面积如下

(6)Area=(x2−x1)(y3−y1)−(x3−x1)(y2−y1)

  • 若 Area>0 → CCW（正面）

  • 若 Area<0 → CW（背面）

可以根据需求不同，我们需要通过配置来决定是否开启剔除，是剔除正面三角形还是背面三角形。比如对于封闭的不透明物体，我们可以开启背面剔除来优化性能，而对于透明物体与双面材质，我们必须关闭背面剔除来获取准确的渲染结果。

4. 光栅化阶段

光栅化阶段接受屏幕空间下的三角形顶点数据，包括顶点的屏幕坐标与其他数据（法线，纹理坐标等），结果三角形设置与三角形遍历的操作后，输出每一个片元的数据交给片元着色器来实现最终的着色。通常由固定光栅化硬件单元（Fixed-Function Rasterizer）来实现，基本不可配置（MSAA与Conservative Rasterization可能会对遍历行为有影响）。

有些资料与书籍将着色阶段也包含在光栅化之中，有的将两者区分。本文采用后者来强调从三角形操作得到片元的过程。

4.1 三角形设置（Triangle Setup）

三角形设置的输入是完整的屏幕空间三角形，每个三角形只发生一次，主要是进行一些准备工作，包括三个方面的内容

4.1.1 计算边方程（Edge Equations）

对三角形的三条边，构建有向边函数（Edge Function）Ei(x,y)=Aix+Biy+Ci ，用于下一步遍历时判断某个片元是否在三角形内部。

4.1.2 计算轴对齐包围盒（AABB）

计算三角形在屏幕上的最小轴对齐包围盒（AABB）[xmin,xmax]×[ymin,ymax] ，从而确定并限制下一步遍历的范围，避免检查整个屏幕。

4.1.3 构建属性数据的平面方程

在光栅化阶段，每个三角形内部顶点的属性值都是由三个顶点利用重心插值得到的。我们可以计算当前点的重心坐标 α,β,γ ，且 α+β+γ=1 ，若顶点属性值记作 attr ，则应有下式成立

(7)attr=α⋅attr0+β⋅attr1+γ⋅attr2

但重心坐标是根据顶在屏幕空间的位置计算得到的，而屏幕空间的坐标在应用了透视除法之后就无法保证线性了，属性值 attr 则在齐次空间裁剪阶段完成后没有发生变化，因此直接应用上面的式子是无法得到准确的结果的。要想得到正确的结果，我们需要利用透视除法时保留的 1w 对重心坐标进行透视校正。

(8)attr=α⋅(attr0/w0)+β⋅(attr1/w1)+γ⋅(attr2/w2)α⋅(1/w0)+β⋅(1/w1)+γ⋅(1/w2)

在上面这个式子中，可以看到我们对分子与分母都用同样的重心坐标进行了插值，分子插值的对象是 attr/w ，分母的插值对象是透视除法的保留值 1w ，可以证明，这样计算可以保证最终得到的属性值是正确的。

在具体实践中，对于插值操作，我们更希望为每一个属性 f 计算得到一个形如 f(x,y)=Ax+By+C 的平面方程，这样不用对每一个顶点都计算一次重心坐标，而是可以用增量算法，这样就更符合计算特性。由于上面的式子对分子与分母都用同样的重心坐标进行了插值，因此我们也需要对分子和分母分别求平面方程 fattri/wi,f1/wi ，然后后续计算时得到 fattr=fattri/wif1/wi 。

如果开发者知道某个属性不需要透视校正（如屏幕空间 UV、调试颜色），可在着色器中标记 noperspective 修饰符。

4.2 三角形遍历（Triangle Traversal）

三角形遍历在上一步输出的包围盒中逐个点或像素块，检查是否满足条件，若是则生成一个片段（Fragment）并计算其相关属性来输出到下一个着色阶段，否则剔除这个点。这一步是光栅化的核心循环，执行代码实例参考如下

xxxxxxxxxx
// 获取包围盒
int left = 0, right = 0, top = 0, bottom = 0;
get_aabb(screen, left, right, top, bottom);
// 遍历包围盒
for (int x = left; x <= right; x++)
{
    for (int y = top; y <= bottom; y++)
    {
        // 计算重心坐标
        vec3 bc = ABC.invert_transpose() * vec3{static_cast<double>(x), static_cast<double>(y), 1.};
        // 剔除不在三角形中的点
        if (bc.x < 0 || bc.y < 0 || bc.z < 0) continue;
        // 插值得到深度z值，进行深度测试，缓存z值
        double z = bc * vec3{ndc[0].z, ndc[1].z, ndc[2].z};
        if (z <= depth[x + y * width]) continue;
        depth[x + y * width] = z;
        // 绘制
        auto frag_result = s.fragment(bc);
        if (frag_result.first) b.set(x, y, frag_result.second);
    }
}

4.2.1 两次遍历

现代的 GPU 在执行三角形遍历时往往会经过两次遍历，第一次遍历是粗粒度遍历（Coarse / Tile-Based Rasterization），他将包围盒区域划分为固定大小的像素瓦片（tile）通常是 8×8、16×16 的像素块，然后遍历所有的Tile确定是否在三角形中，从而可以跳过不相关的像素块。第二次遍历是细粒度遍历（Fine Rasterization），在相交的 Tile 内，以 2×2 的像素块（Quad） 为基本单位遍历来判断是否在三角形内，这样有利于抗锯齿和 SIMD 优化。

对每个点 (x,y) ，通过计算并验证 E0(x,y)≥0,E1(x,y)≥0,E2(x,y)≥0 。若全部满足，则说明像素在三角形内部（或边上）。现代实现使用增量算法（Incremental Evaluation）避免重复乘法。

4.2.2 顶点属性值计算

在三角形设置过程中，我们已经得到了各个属性的平面方程，这里代入各个点的坐标计算即可，这里同样使用增量算法进行处理。我们假设从左到右、从上到下的顺序遍历三角形内的像素，计算属性值 f ，初始值 f=Ax+By+C ，接下来向右移动，坐标从 (x,y) 变为 (x+1,y) ，则 fx+1=A(x+1)+By+C=Ax+By+C+A=f+A ，同理向下移动可得 fy+1=f+B。这样非常适合硬件计算，利于SIMD优化。

 

4.2.3 Early-Z

插值得到深度值 zfrag 与深度缓冲比较（若启用 Early-Z 且条件满足）失败则丢弃整个 Quad（节省 4 个片元的着色开销）。这样可以确保大量被遮挡的片段在着色前就被剔除，显著提升性能。但是 Early-Z 与透明物体的渲染会发生冲突（ Early-Z 确定可见性时颜色未知，而透明物体一定要计算得到颜色才能确定可见性），因此当后续片元着色器使用discard或开启混合时，Early-Z 往往会失效。

三角形设置与三角形遍历的分离可以避免每个像素重复计算边方程和插值系数，同时三角形设置可在专用硬件单元完成，三角形遍历可在大量 ALU 上并行。

5. 着色阶段

着色阶段的操作对象是片元（Fragment），这一阶段主要接受光栅化阶段输出的一系列数据，通过特定的计算与测试来最终决定出出这个片元到底要显示成什么颜色。

5.1 片元着色器（Fragment Shader）

片元着色器也叫做像素着色器（Pixel Shader），除了光栅化阶段输出的顶点数据以外，作为可编程着色，片元着色器还可以从纹理显存、常量寄存器中读取其他数据。片元着色器是光栅化渲染管线的核心工作，他可以通过着色模型的计算来计算出对应片元的颜色。常见的着色模型参考如下

由于不同的着色模型在实现的过程中所需的数据也不同，因此之前顶点的属性与纹理等数据准备往往都是在确定着色模型后，为了片元着色器的最终计算而服务的。不同的着色模型往往意味着整个渲染程序的巨大差异，因此可以说着色模型的实现是现代光栅化程序的核心。

5.2 测试与混合（Testing & Blending）

在片元着色器输出后片元计算结果后，在写入帧缓冲区之前，还需要进行一个关键的固定功能阶段，也就是测试与混合阶段。这个阶段是高度可配置的，每个阶段都可以配置为是否启用以及如何执行，他决定一个片元（Fragment）是否能最终写入屏幕（或帧缓冲）中。

模版测试之前其实还存在一个透明度测试，用以剔除完全透明的着色结果。但由于与 Early-Z 的优化冲突，现代的渲染管线几乎全部将他弃用了，现在可以通过在片元着色器中判断 alpha 通道是否大于 0 来达到同样的效果。

xxxxxxxxxx
if (color.a < 0.1) discard; // 丢弃完全透明的片段

5.2.1 模版测试（Stencil Testing）

模版测试会读取每个片元对应的模版值，将其与模版缓冲中的值进行比较（也可以通过通过参考值、掩码等进行比较），通过后才将片元发给下一步的深度测试。利用模版测试可以实现复杂遮挡逻辑，如镜面反射区域、轮廓描边、阴影体积等。

5.2.2 深度测试（Depth Testing）

深度测试通过比较当前片元的深度值 zfrag 与深度缓冲区中的深度值 zbuffer 来决定是否进行绘制，如果测试通过则保留，还会更新深度缓冲中的值。可以通过配置比较函数来决定保留远处的值还是近处的值，以适应不同的需求。参考的实现代码如下

xxxxxxxxxx
if (z <= depth[x + y * width]) continue;
depth[x + y * width] = z;
// 发送到缓冲区 ......

5.2.3 混合（Blending）

混合决定了当前片元的颜色值如何写入缓冲区，可以写成这种形式 Colorbuff=F(Colorfrag,Colorbuff) ，如果不开启混合，也就是默认行为下，片元颜色值会直接覆盖当前的缓冲区中的颜色值： Colorbuff=Colorfrag ，开启后可以以多种方式来进行混合操作，比如按权重加权 Colorbuff=α⋅Colorfrag+(1−α)⋅Colorbuff 。这正是 Alpha 混合（Alpha Blending） 的标准形式，用于实现玻璃、烟雾、UI 透明等效果。混合操作的流程顺序参考如下

这里操作流程基本就是“读、改、写“，因此混合操作依赖于帧缓冲区的当前状态，这句要求多个半透明物体必须按从后往前排序才能正确渲染（否则会出现错误叠加）。

5.3 其他后处理

在完成所有计算与测试之后，我们得到了完整的一个帧缓冲，如同前面介绍的，帧缓冲本质上就是一张位图，因此我们可以借助图像处理的方式，低成本实现原本需要复杂计算与成本才能达到的视觉效果。这一后处理操作通常是基于屏幕空间坐标来完成，因此可以通过屏幕空间坐标来读取其他数据。常见的后处理效果介绍如下

  • 抗锯齿

  • 光照效果

  • 色彩校正与色调映射

  • 动态模糊：基于时序与 Motion Vector 模拟相机或物体高速运动时的模糊效果

  • SSAO（屏幕空间环境光遮蔽）：基于法线与深度估计来增强物体接触处的阴影，提升立体感

传统的后处理操作是将整个帧缓冲区作为一个纹理，通过渲染屏幕矩形来实现后处理的功能。而现代图形接口也可以使用计算着色器（Compute Shader）来实现这些高度并行的计算需求。

6. 屏幕显示

当我们完成以上所有操作后，我们终于可以将存储了渲染结果的帧缓冲显示在屏幕上了。这一步由硬件与操作系统决定，这里需要关注设备的像素比（Device Pixel Ratio），因为我们的渲染结果输出的是逻辑的像素单位，而最终屏幕上显示的内容与结果是物理的像素单位，因此最终的输出结果有可能会因此发生偏差（如 Apple 的 Retina 显示屏通常 DPR = 2 ，真实的 Frame 尺寸可能是实际的两倍）。
