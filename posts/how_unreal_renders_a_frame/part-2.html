发布时间：2017 年 10 月 25 日
原文分析版本：Unreal Engine 4.17.1

这是“How Unreal Renders a Frame”系列的第二篇，另见 Part 1 和 Part 3。

这一篇继续分析 Unreal 如何渲染一帧，内容包括 light grid generation、G-Prepass 和 lighting。

1. Light Assignment（灯光分配）

renderer 接下来切换到 Compute Shader，在 ComputeLightGrid pass 中把 light 分配到一个 3D grid，做法与 Clustered Shading 相近。根据 surface 的位置查询 light grid，便能快速找出影响它的 light。

从 pass 名称可以看出，view space 下的 light grid 尺寸为 29×16×32。Unreal 在 screen space 使用 64×64 pixel 的 tile，Z 方向划分为 32 层，因此 light grid 的 X、Y 维度会随屏幕分辨率变化。名称还显示，此处要分配 9 盏 light 和 2 个 Reflection Probe。Reflection Probe 是带有位置和半径的“实体”，负责捕获周围环境，供场景物件计算 reflection。

根据 Compute Shader LightGridInjection.usf 的源码，Z 方向使用 exponential partition，view space 中的 cell 会随距离增加而变厚。每个 cell 以 Axis-Aligned Bounding Box 与 light bounding volume 做 intersection test。light index 先存入 linked list，再在 Compact pass 中转换成连续数组。

后续的 Volumetric Fog pass 会使用这份 light grid 向 fog 中加入 light scattering；Environment Reflection pass 和 Translucency pass 也会读取它。

CullLights pass 中还有一个有意思的细节。它一开始要清空存储 light data 的 Unordered Access View（UAV），三个 UAV 中只有两个使用 ClearUnorderedAccessViewUint。剩下的一个由 Compute Shader 手动写入清零，也就是上面列表中的第一次 Dispatch。从源码看，buffer 超过 1024 byte 后，Unreal 更倾向于用 Compute Shader 清空，而不是调用 clear API。

2. Volumetric Fog（体积雾）

接下来计算 Volumetric Fog，这部分仍然使用 Compute Shader。

这个 pass 把 transmittance 和 light scattering 计算并保存到 volume texture。之后只需根据 surface position 采样，便能得到 fog。和前面的 Light Assignment 类似，这个 volume 会拟合 view frustum：每个 tile 覆盖 8×8 pixel，深度方向有 128 个 slice。depth slice 按 exponential distribution 排列，同时把 near plane 稍微向远处推，避免相机附近出现大量很薄的 cell。这个做法与 Avalanche 的 Clustered Shading 系统相近。

与 Assassin's Creed IV 和 Frostbite 的 Volumetric Fog 技术类似，fog 分三个 pass 计算。

第一个是 InitializeVolumeAttributes，它计算 scattering 和 absorption 等 fog parameter，写入 volume texture，同时把全局 emissive value 写入第二张 volume texture。

第二个是 LightScattering。它把带 shadow 的 Directional Light、Sky Light 和 local light 合并起来，计算每个 cell 的 light scattering 与 extinction。这些 local light 已经在前面的 ComputeLightGrid pass 中分配到 light volume。Compute Shader 输出的 Light Scattering 和 Extinction 还会经过 Temporal Anti-Aliasing（TAA）；history buffer 本身也是一张 3D texture，用于逐帧改善每个 grid cell 中的 scattered light 质量。

最后的 FinalIntegration 沿 Z 方向 ray march 这张 3D texture，逐步累积 scattered light 与 transmittance，并把途中得到的结果写入对应 cell。

3. G-Prepass

随后执行的是 Unreal 版本的 G-Prepass，这类 pass 常见于 Deferred Rendering 架构。它把 material property 缓存到一组 render target，减少后续昂贵 lighting 和 shading 计算中的 overdraw。

这个 pass 通常会渲染所有 solid prop，包括 Static 和 Movable 物件。Unreal 甚至会先画 skydome。大多数情况下，这种顺序会浪费 GPU 工作，因为离相机更近的物件随后会覆盖 skydome。不过前面执行过 Z-Prepass，此时 skydome 的 overdraw 已被消除；至少对 solid prop 而言，场景中的大部分 overdraw 也一并消失了。

下图是 G-Prepass 写入的 render target 列表。

depth buffer 此时只绑定给 Z-Test 使用。它已经在 Z-Prepass 中写满，renderer 不会再次写入 depth。stencil buffer 仍然可写，用来标出 solid geometry 覆盖的 pixel。

G-Buffer 的内容会随 rendering setting 改变。例如 renderer 如果要把 velocity 写进 G-Buffer，GBufferD 会被 velocity 占用，其他数据也要跟着调整。在当前场景和 rendering path 下，G-Buffer layout 如下。

SceneColorDeferred：保存 Indirect Lighting。

GBufferA：保存 world space normal，format 为 RGB10A2_UNORM，看起来没有使用特殊 encoding。

Distortion：保存多种 material property，包括 metalness、roughness、specular intensity 和 shading model。

GBufferC：RGB 保存 Albedo，Alpha 保存 Ambient Occlusion（AO）。

GBufferE：保存由 shading model 决定的 Custom Data，例如 subsurface color 或 tangent vector。

GBufferD：保存预烘焙的 shadowing factor。

Stencil：标记 solid prop。

场景中除 Movable Rock 和 skydome 之外，所有 solid prop 都从三张带 mipmap 的 atlas 中采样 lighting data。这些 atlas 看起来分别缓存了 irradiance、shadow 和 surface normal。

4. Particle Simulation Revisited（再次模拟粒子）

一帧最开始的 Particle Simulation 已经写出了 particle sprite 的 world position 和 velocity。由于当时执行得太早，renderer 还拿不到 depth buffer 和 normal buffer，无法在 GPU 上处理 collision。现在这些 buffer 已经就绪，需要 collision 的 particle 会重新执行一次 simulation。

5. Velocity Rendering（速度渲染）

Unreal 默认把移动物件的 velocity 写入单独的 R16G16 buffer。后续的 Motion Blur 以及任何需要 reprojection 的效果，例如 Temporal Anti-Aliasing，都会使用这份 velocity。当前场景只有 Rock 被标记为 Movable，因此 velocity buffer 中也只有它。

6. Ambient Occlusion（环境光遮蔽）

material information 准备完毕后，renderer 即将进入 lighting 阶段。在此之前，先要计算 Screen Space Ambient Occlusion（SSAO）。

场景中没有 deferred decal。我猜如果加入 decal，上图中空的 DeferredDecals pass 会修改 G-Buffer 里的部分 material property。

SSAO 分两个 pass 计算，一个处理 quarter-resolution buffer，另一个处理 full-resolution buffer。AmbientOcclusionPS 908×488 使用 AmbientOcclusionSetup pass 生成的低分辨率 normal buffer、前面生成的 Hi-Z buffer，以及一张 random vector texture，对 depth 和 normal buffer 采样。Shader 每帧采样 random vector texture 时还会加入 jitter，以模拟 supersampling，让 AO 质量随时间逐步改善。

随后，AmbientOcclusionPS 1815×976 计算分辨率更高的全屏 AO，并与低分辨率结果合并。最终质量相当不错，也不需要额外的 blur pass。

full-resolution AO buffer 最后会作用到 SceneColorDeferred。这张 buffer 属于前面的 G-Buffer，目前保存的是场景的 indirect lighting，也就是 ambient lighting。

7. Lighting（光照）

在分析 solid prop 的 lighting 之前，先说一下 Unreal 如何给 translucent surface 打光，后面会多次遇到这个系统。Unreal 把 light 注入两张 64×64×64、format 为 RGBA16_FLOAT 的 volume texture。其中 TranslucentVolumeX 保存到达每个 volume cell 的 light，以 Spherical Harmonics 表示，已经包含 shadow 和 attenuation；TranslucentVolumeDirX 保存各个 light source 的近似方向。

renderer 维护两组这样的 texture。一组覆盖相机附近的物件，lighting 分辨率较高；另一组覆盖远处物体，分辨率可以低一些。空间分配思路与 Cascaded Shadow Map 相近，离相机越近，分到的 texel 越多。

下面是近距离 cascade 的 translucency lighting volume，其中只注入了带 shadow 的 Directional Light。

这些 translucency lighting volume 不会影响 solid prop。后面渲染 translucent prop 和 particle effect 时才会用到，不过 volume 本身会在当前 lighting pass 中写入。

回到 solid prop 的 direct lighting。renderer 现在可以开始计算 lighting，并把结果应用到场景中。draw call 列表的长度取决于 light source 数量，下图只展开了最相关的部分。

light 分成 NonShadowedLights 和 ShadowedLights 两组处理。NonShadowedLights 包括 particle effect 使用的 simple light，以及不投射 shadow 的普通 scene light。两者的区别之一是，普通 scene light 渲染时会做 Depth Bounds Test，避免给近似 light volume 之外的 pixel 计算 lighting。这个功能通过特定 driver extension启用，结果累积到前面提过的 SceneColorDeferred。

另一个区别是，simple light 看起来完全不会写入 translucency lighting volume。renderer 源码似乎预留了相关逻辑，可能有某个暴露出来的 setting 可以控制它。

如果场景中可见、无 shadow 且非 Static 的 light 超过 80 盏，renderer 会从传统 Deferred Shading 切换到 Tiled Deferred Lighting。

此时 Compute Shader 会计算这些 light 的 lighting，light data 通过 constant buffer 传给 Shader。感谢 wand de 指出这一点。切换到 Tiled Deferred 并在一个 Compute Shader pass 中应用所有 light，看起来只改变 direct lighting 的处理。InjectNonShadowedTranscluscentLighting pass 仍会逐盏把 light 注入 translucency lighting volume，每盏 light 都对应一个单独的 draw call。

ShadowedLights pass 处理所有投射 shadow 的 light，包括 Stationary 和 Movable。默认情况下，每盏 shadow-casting light 分三个阶段处理。

首先，ShadowProjectionOnOpaque 计算 screen space shadow；接着，InjectTranslucentVolume 把 light contribution 注入 translucency lighting volume；最后由 StandardDeferredLighting 计算场景 lighting。

对于当前场景的 Directional Light，只有 Split0 含有 shadow information，前文已经提到过。shadow 计算结果写入一张屏幕大小的 RGBA8 buffer。

下一步的 InjectTranslucentVolume 会把 Directional Light 的 contribution 写入前面介绍的两个 cascade。每个 InjectTranslucentVolume pass 因而有两个 draw call。最后，StandardDeferredLighting 在 screen space shadow buffer 的遮罩下计算 lighting，并把结果应用到 SceneColorDeferred。

local light 似乎也遵循同样的流程：先把 shadow 投影到 screen space buffer，再向 translucency lighting volume 注入 light，最后把 lighting 累积到 SceneColorDeferred。

Movable 和 Stationary local light 的处理大体相同。区别在于 Movable light 会把带 shadow 的 lighting 注入 translucency lighting volume；在 shadow 资源上，投射 shadow 的 Movable light 使用 cubemap，而不是 per-object atlas。

所有 light 共用同一张 screen space shadow buffer render target。处理每盏 light 时，只清空它会写入的相关区域，我猜这是为了节省显存。

lighting pass 结束后，SceneColorDeferred 已经包含场景中全部 direct light 的累积结果。

renderer 虽然在前面的 Light Assignment pass 中生成了用于 light binning 和 clustering 的数据结构，但 solid geometry 的 lighting 完全没有使用它。这里采用的仍是较传统的 Deferred Shading 路径，每盏 light 单独渲染。

最后，renderer 对两个 cascade 的 translucency lighting volume 做 filter，减轻 translucent prop 和 effect 在 lighting 时出现的 aliasing。
