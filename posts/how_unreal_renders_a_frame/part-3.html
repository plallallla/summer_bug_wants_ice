1. Image Space Lighting（图像空间光照）

接下来计算全屏 Screen Space Reflection（SSR），render target format 为 RGBA16_FLOAT。

Shader 还会使用一帧开始阶段生成的 Hi-Z buffer 加速 intersection。ray march 时，它根据 surface roughness 选择 Hi-Z mip。surface 越粗糙，reflection 中能看清的细节越少，ray tracing 的步进也就可以更粗。每一帧还会对 ray 的起点施加 jitter，再结合 Temporal Anti-Aliasing 提高 reflection 的图像质量。

ray march 找到 hit 后，Shader 从上一帧的 render target 采样 color。reflection 里能看到 Volumetric Fog 和透明雕像，右侧椅子下方还能隐约看到 particle effect。transparent surface 没有准确的 depth，无法求出正确 hit，reflection 因而经常被拉长。不过在不少场景中，最终效果仍然很可信。

ReflectionEnvironment pass 使用 Compute Shader 把 SSR 应用到主 render target，同时叠加场景中两个 Reflection Probe 捕获的 Environment Reflection。每个 probe 的 reflection 都保存在一张带 mipmap 的 cubemap 中。

Environment Reflection Probe 在游戏启动时生成，只捕获 Static geometry。上面的 cubemap 中没有参与动画的 Rock，原因就在这里。

2. Fog and Atmospheric Effects（雾与大气效果）

如果场景启用了 fog 和 atmospheric effect，接下来便轮到它们。

renderer 首先生成一张 quarter-resolution light shaft occlusion mask，标出哪些 pixel 会接收 light shaft。在这个场景中，light shaft 只来自 Directional Light。

随后用 Temporal Anti-Aliasing 改善 mask 质量，并连续执行三个 blur pass，得到下图的 mask。原图几乎全白，我对它做了增强，才看清其中的内容。

只看这次 GPU capture，我还不清楚为什么要先对 mask 做 Temporal AA，再执行 blur，因为最终结果的分辨率很低。要弄清这个选择，可能还得观察更多环境和使用场景。

把 fog 和 light shaft 应用到场景之前，renderer 暂时转去处理 Atmospheric Scattering，把结果直接写入 full-resolution 主 render target。

这看起来是一套完整的 scattering 计算，使用预计算的 transmittance、irradiance 和 inscattering，与 Bruneton 等人的方法相似。

可惜这是一个室内场景，simulation 的效果不太明显。

最后，renderer 把 Exponential Fog 和 light shaft 应用到场景。

Shader 使用前几个 pass 生成的 Volumetric Fog volume texture，并根据 solid geometry 的位置采样，同时叠加前面计算出的 light shaft mask。

3. Transparency Rendering（透明渲染）

solid prop 的 fog 处理完成后，renderer 开始绘制 translucent geometry 和 effect。

场景中有两座玻璃雕像。它们最先渲染，使用普通 Alpha Blending 混合到主 render target。

两件 transparent prop 和场景融合得很好，会受到 local light、Directional Light、Environment Reflection 和 fog 等效果影响。默认情况下，renderer 使用高质量 Shader 绘制 transparent prop。它会采样 Atmospheric Simulation 的预计算 texture、烘焙 lightmap data、保存 Directional Light 与 local light 的 translucency lighting volume，以及 Reflection Probe cubemap，再据此计算 lighting。

不过我没有看到 Shader 读取 Volumetric Fog volume texture。它似乎只计算与高度和距离有关的 fog，也可能是我漏掉了某个 setting。与 Atmospheric Scattering 一样，distance fog 在 Vertex Shader 中计算。

particle effect 会写入一张单独的 full-resolution render target。

与 transparent prop 一样，particle 的 Atmospheric Scattering 和 fog 也在 Vertex Shader 中计算。根据 Particle System setting，renderer 还可以用 translucency lighting volume 给 particle 打光；我见到的那次调用是在 Pixel Shader 中完成的。

transparency 结束前，renderer 还要用一个 pass 计算 refraction。

transparent prop 和启用了 refraction 的 particle 会再渲染一次，向一张 full-resolution buffer 写入 distortion vector，供后面计算 refraction。为了让 vector 更容易看清，我增强了下图。这个 pass 同时启用 stencil buffer，标记需要 refraction 的 pixel。

计算 refraction 的 DistortionApply pass 读取当前主 render target 和 distortion vector，写出一张看起来有些奇怪的 refraction texture。

stencil buffer 已经标出了要接收 refraction 的 pixel，因此 renderer 不必先清空整张 texture。

最后一个 refraction pass 根据 stencil，把 refraction texture 复制回主 render target。

右侧椅子上或许能看出 refraction，那是尚未合入场景的 particle 造成的。对于 transparent prop，refraction 在 prop 本身绘制之后执行。

下一个 BokehDOFRecombine pass 终于把 particle 合入场景。这个 Shader 很简单，做的事没有 pass 名字听起来那么多，也可能是 rendering setting 不同造成的。

4. Post Processing（后处理）

一帧时间线的最后一段由若干 Post Processing pass 组成，下面简要走一遍。

按照当前场景配置，renderer 会对主 render target 依次执行 Temporal Anti-Aliasing、Motion Blur、Auto Exposure、Bloom 和 Tonemapping。

Unreal 的 Temporal Anti-Aliasing 使用 history buffer 随时间累积 sample，渲染过程分为两个 pass。第一个 pass 处理没有 stencil 标记的 pixel，这里对应一部分 particle。它读取主 render target、history buffer 和 velocity buffer，通过 reprojection 计算 Temporal AA。

接着用类似的 Temporal AA pass 处理有 stencil 标记的部分，生成最终的 anti-aliased image。

两个 pass 的差别在 blend factor，也就是 feedback。第一个 pass 在 history buffer 和当前 render target 之间使用可变 blend factor；具体数值可随 pixel luminance、距离和 renderer 提供的 weight 等参数变化。第二个 pass 使用固定的 0.25，最终 pixel 主要来自当前 sample。我猜这是为了减轻快速运动 particle 的 ghosting，因为我们没有它们的 velocity information。

接下来是 Motion Blur。在它之前，renderer 先执行 velocity flattening 和 dilation pass。

这个场景中的 Motion Blur 不太明显。相机没有移动，唯一拥有 velocity 的移动物件只有 Rock，而它已经因为运动和 Temporal Anti-Aliasing 略微变模糊了。

为了实现 Auto Exposure，也就是 Eye Adaptation，renderer 用 Compute Shader 根据当前场景的 luminance 生成 histogram。histogram 会把 pixel intensity 分 bin，并统计落入每个 intensity bin 的 pixel 数量。

有了 histogram，renderer 可以方便地跳过过暗或过亮的区域，更合理地估计场景平均 luminance。然后根据平均值调整 exposure，完成 Eye Adaptation：画面越亮，exposure 越低；画面越暗，exposure 越高。

随后是一系列使用 Gaussian Filter 的 downscale pass，再通过若干 upscale 与 combine 操作实现 Bloom。下图关闭了 exposure control，并做过调整，让 Bloom 更容易观察。

PostProcessCombineLUTs pass 使用 Geometry Shader 和一段相当长的 Pixel Shader，生成 Tonemapping 阶段要读取的 Color Grading LUT。LUT 是一张 32×32×32、format 为 RGB10A2 的 volume texture。

一帧的最后一个 pass 是 Tonemapper。它把前面计算的 Bloom 与主 render target 合并，使用 Eye Adaptation 的结果调整 exposure，再让 color 通过 Color Grading LUT，得到最终 pixel color。

5. 总结

这里必须再次说明，我们看到的只是 renderer 中的一条 path。许多 parameter 和 setting 都会改变执行过程，这次分析其实只触及了表面。

整个过程很有意思，不过一帧中的工作量实在太大，文章最后更接近在回答 renderer“做了什么”，没有像我原先希望的那样充分解释“怎么做”。不少内容仍未深入，我以后还想回来继续研究。

Unreal renderer 的源码没有大面积文档，不过代码本身相当整洁，也容易理解。沿着 draw call 列表查找，通常能很快找到对应实现。Shader 则难跟踪得多，因为源码大量使用 conditional compilation。要是能有一份中间 cache，保存已经处理、可以编译的 Shader specialization，并把它们的名字注入 draw call 列表，检查和 performance profiling 都会方便不少。

Unreal renderer 的默认配置明显偏向高质量画面。环境、lighting 和 volumetric 等数据会尽可能预烘焙，同时充分利用 Temporal Anti-Aliasing 改善图像质量。

场景物件很多、又缺少有效 occluder 时，需要留意 occlusion pass 的开销。transparent prop 和 particle 开启 refraction 后必须渲染两次。大量 Stationary 或 Movable local light 也会增加 Lighting pass 的成本，因为它们逐盏渲染，同时还会增加 Translucency 和 Volumetric Lighting 的 light injection 开销。
